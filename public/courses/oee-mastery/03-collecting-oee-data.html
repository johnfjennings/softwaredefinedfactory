Collecting OEE Data

OEE is only as good as the data behind it. Garbage in, garbage out. This lesson covers the practical side of data collection — what to capture, how to capture it, and the common pitfalls that lead to misleading OEE numbers.

The Five Data Points You Need

At minimum, calculating OEE requires five pieces of information per production run:

  1. Planned production time — How long was the equipment scheduled to run?
  2. Downtime events — When did the equipment stop, for how long, and why?
  3. Ideal cycle time — What is the theoretical best speed for this product?
  4. Total output — How many units were produced (good + defective)?
  5. Defect count — How many units failed to meet quality specifications?

Key Insight: Most factories already track output and defects. The challenge is accurate downtime data — knowing exactly when equipment stopped and categorizing the reason correctly.

Manual vs. Automated Data Collection

Manual Collection

Operators record data on paper forms or spreadsheets. This is how most manufacturers start.

Pros:

  • Zero upfront investment
  • Forces operators to engage with the data
  • Builds awareness of losses before investing in automation

Cons:

  • Inaccurate — operators estimate downtime duration and often round
  • Incomplete — short stops under 2-3 minutes are rarely recorded
  • Delayed — data is often entered at end of shift from memory
  • Time-consuming — pulls operators away from value-adding work

Semi-Automated Collection

Equipment signals (run/stop) are captured automatically, but operators add context (reason codes, notes).

How it works:

  • A sensor or PLC signal detects when the machine is running
  • The system automatically records start/stop timestamps
  • When a stop occurs, the operator selects a reason from a dropdown on a tablet or HMI
  • Output counts come from machine counters or sensors

Best Practice: Semi-automated is the sweet spot for most manufacturers. You get accurate timing with human context — the "what happened" that pure automation misses.

Fully Automated Collection

MES (Manufacturing Execution Systems) or IIoT platforms capture everything automatically with minimal operator input.

  • Machine states detected in real-time via PLC/SCADA integration
  • Reason codes auto-classified using machine learning patterns
  • Part counts from vision systems or encoder signals
  • Quality data from inline inspection systems

Best for: High-volume, high-speed lines where manual recording is impossible (e.g., packaging at 500+ units/minute).

Defining Your Ideal Cycle Time

This is the most commonly misunderstood OEE input. The ideal cycle time should be:

Use This Not This
The fastest sustainable speed the equipment can run for this specific product The average speed you typically run at
Based on nameplate capacity or time studies Based on "what feels right" or historical averages
Product-specific (different products may have different ideal speeds) One number for all products

Common Mistake: Setting the ideal cycle time too slow inflates your OEE and hides speed losses. Setting it unrealistically fast makes OEE look impossibly low. Use the manufacturer's rated speed or your best-ever sustained run as the benchmark.

Downtime Reason Codes

A well-designed reason code system is critical. Keep these principles in mind:

  • Keep it short: 10-20 codes maximum. Too many codes and operators won't use them correctly.
  • Make codes specific enough to act on: "Mechanical failure" is too vague. "Bearing failure — Drive motor" is actionable.
  • Map codes to the Six Big Losses: Each reason code should clearly fall under one of the six categories.
  • Include an "Other" code with a free-text field — but monitor it. If "Other" exceeds 10%, your code list needs updating.
  • Separate planned from unplanned stops: Changeovers, scheduled maintenance, and breaks are planned. Breakdowns are unplanned.

Example Reason Code Structure

Code Description Loss Category
BD-01 Mechanical breakdown Breakdown (Availability)
BD-02 Electrical failure Breakdown (Availability)
CO-01 Product changeover Setup (Availability)
CO-02 Material change Setup (Availability)
MS-01 Jam / blockage Minor Stop (Performance)
MS-02 Sensor trip Minor Stop (Performance)
QD-01 Out-of-spec product Defect (Quality)
QD-02 Startup rejects Startup Loss (Quality)

Data Quality Pitfalls

Watch out for these common problems that corrupt OEE data:

  1. Rounding downtime: "About 10 minutes" recorded for a 17-minute stop. Use timestamps, not estimates.
  2. Missing short stops: Stops under 2-3 minutes often go unrecorded in manual systems. These add up.
  3. Misclassifying planned vs. unplanned: Calling an unplanned breakdown a "planned" stop inflates Availability artificially.
  4. Incorrect part counts: Double-counting, missed rejects at downstream inspection, or counter resets mid-shift.
  5. Ignoring warm-up time: Some machines take 15-30 minutes to reach operating temperature. Is this planned or unplanned?

Rule of Thumb: Start with manual data collection for 2-4 weeks before investing in automation. You'll learn what matters, what's hard to capture, and what your operators need — all before spending any money on sensors or software.
